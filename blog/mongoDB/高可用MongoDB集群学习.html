        # NoSQL的特征

        ​ MongoDB是一个介于关系数据库和非关系数据库之间的产品，是非关系数据库当中最像关系数据库的。其特征NoSQL、文档存储、Json数据模型。

        高扩展性：Nosql去掉了关系数据库的关系型特性，易于横向扩展。

        高性能：Nosql通过简单的key-value方式获取数据，非常快速。还有NoSQL的Cache是记录级的，是一种细粒度的Cache，所以NoSQL在这个层面上来说就要性能高很多。

        灵活的数据模型：相对于关系数据库里，增删字段的繁琐，NoSQL无需事先为要存储的数据建立字段，随时可以存储自定义的数据格式。

        高可用：NoSQL在不太影响性能的情况，就可以方便的实现高可用的架构。比如mongodb通过mongos、config server replica set，shard就可以快速配置出高可用配置。

        # Mongodb数据备份与恢复

        # mongodb数据备份

        备份命令脚本语法如下：

        mongodump -h dbhost -d dbname -o dbdirectory
        1
        • -h： MongDB所在服务器地址，例如：127.0.0.1，当然也可以指定端口号：127.0.0.1:27017
        • -d： 需要备份的数据库实例，例如：test
        • -o： 备份的数据存放位置，例如：c:\data\dump，当然该目录需要提前建立，在备份完成后，系统自动在dump目录下建立一个test目录，这个目录里面存放该数据库实例的备份数据。

        进入MongoDB安装目录的bin目录输入命令mongodump，执行以上命令后，客户端会连接到ip为 127.0.0.1 端口号为 27017 的MongoDB服务上，并备份所有数据到 bin/dump/ 目录中。命令输出结果如下：

        mongodump 命令可选参数列表如下所示：

        语法 描述 实例
        mongodump --host HOST_NAME --port PORT_NUMBER 该命令将备份所有MongoDB数据 mongodump --host runoob.com --port 27017
        mongodump --dbpath DB_PATH --out BACKUP_DIRECTORY mongodump --dbpath /data/db/ --out /data/backup/
        mongodump --collection COLLECTION --db DB_NAME 该命令将备份指定数据库的集合。 mongodump --collection mycol --db test

        # MongoDB数据恢复

        mongodb使用 mongorestore 命令来恢复备份的数据。

        恢复命令脚本语法如下：

        >mongorestore -h <hostname><:port> -d dbname <path>
        1

        • --host <:port>, -h <:port>：

          MongoDB所在服务器地址，默认为： localhost:27017

        • --db , -d：

          需要恢复的数据库实例，例如：test

        • --drop：

          恢复的时候，先删除当前数据，然后恢复备份的数据。就是说，恢复后，备份后添加修改的数据都会被删除，慎用哦！

        • <path>：

          mongorestore 最后的一个参数，设置备份数据所在位置，例如：c:\data\dump\test。

          你不能同时指定 <path>和--dir选项，--dir也可以设置备份目录。

        • --dir：

          指定备份的目录

          你不能同时指定 <path>和 --dir 选项。

        同上，进入MongoDB安装目录的bin目录输入命令mongorestore

        # 主从模式

        ​ 主从复制MongoDB中比较常用的一种方式，如果要实现主从复制至少应该有两个MongoDB实例，一个作为主节点负责客户端请求，另一个作为从节点负责从主节点映射数据，提供数据备份，客户端读取等，推荐一主多从模式。

        实现方式：

        • 主节点的操作会被记录为oplog,存储在系统数据库local的集合oplog.$main中，这个集合中的每个文档都代表主节点的一个操作（不包括查询）
        • 从节点定期从主服务器获取oplog数据，并在本机进行执行
        • oplog使用的是固定集合，随着操作的逐渐增加，新的文档会逐渐覆盖旧的文档

        注意点或缺点：

        • 每个从节点必须知晓其对应的主节点地址
        • 只允许主节点进行数据更新操作
        • 主节点宕机服务不可用
        • 3.2版本以后已经移除

        # 副本集

        主从模式存在几个问题：

        • 主节点挂了不能自动切换连接，目前需要手工切换。
        • 主节点的读写压力过大
        • 从节点每个上面的数据都是对数据库全量拷贝，从节点压力会过大
        • 数据压力大到机器支撑不了的时候能否做到自动扩展

        在主从模式下，如果主数据库宕机，从数据库无法自动接管主数据库，从而无法接入数据，取而代之的就是MongoDB副本集模式，

        官方说明：https://docs.mongodb.com/master/core/master-slave/

        ###概述

        ​ mongodb的副本集相当于具有自动故障恢复的主从集群，主从集群和副本集最明显的特征为副本集没有固定的“主节点”，整个集群会通过一定的算法选举出主节点。实际上副本集就是一组mongod维护相同数据集的实例，副本集包含多个数据承载节点和可选的一个仲裁节点，在承载数据的节点中，一个且仅一个成员被视为主节点，而其他节点被视为次要节点。

        **副本集特征： **

        • N 个节点的集群
        • 任何节点可作为主节点
        • 所有写入操作都在主节点上
        • 自动故障转移
        • 自动恢复

        ###副本集介绍

        节点介绍：

        1. 主节点：primary：

        • 接收和处理所有写操作

        • 讲写操作都记录到oplog中

        • 不做特别设定的话，主节点同样会处理所有的读操作

        • 如果主节点不可达，则从节点会发起选举，产生新的主节点

        1. 从节点：secondaries

        • 从节点获取主节点的oplog，并在自身节点进行执行，实现数据同步

        • 经过设置，从节点可以接收读操作

        • 从节点可以设置永远都不成为主节点

        • 可以设置为隐藏节点，对应用不可见

        1. 仲裁节点：arbiter

        • 不会从主节点同步数据，因此也无法成为主节点

        • Arbiter节点始终都有一票投票权

        • 由于不存储数据，所以仲裁节点可以用较少的成本进行故障转移

        • 投票节点最好不要跟主节点和从节点在一台机器

        复制原理：主节点记录在其上的所有操作oplog，从节点定期轮询主节点获取这些操作，然后对自己的数据副本执行这些操作，从而保证从节点的数据与主节点一致。

        ​ 可以将一个额外的mongod实例添加到副本集作为 仲裁者。仲裁者不维护数据集。仲裁者的目的是通过响应其他副本集成员的心跳和选举请求来维护副本集中的仲裁。因为它们不存储数据集，所以仲裁器可以是提供副本集仲裁功能的好方法，其资源成本比具有数据集的全功能副本集成员更便宜。如果您的副本集具有偶数个成员，请添加仲裁者以获得主要选举中的大多数投票，仲裁节点不作为数据存储用，只是在故障转移(选举)的时候起作用，在这种情况下，仲裁节点的机器配置就可以不用那么好了，且仲裁节点不能成为从节点或主节点

        ​

        ​ 官方建议的最低配置是所有节点加起来至少要有三个：一主二从或者一主一从一仲裁。复制集所有节点加起来最多可以有50个节点，但是最多只能有7个投票节点。

        Bully算法

        ​ 如果副本集中主节点宕掉后，需要使用bully算法进行选举主节点，其主要思想为每个成员均可以声明自己为主节点并通知其他节点，别的节点可以选择接受这个声明或是拒绝并进入主节点竞争，只有被其他节点接受的节点才可以当主节点，

        ​ 节点按照一些属性来判断谁应该胜出。这个属性可以是一个静态ID，也可以是更新的度量像最近一次事务ID（最新的节点会胜出）

        官方描述：

        1. 得到每个服务器节点的最后操作时间戳。每个mongodb都有oplog机制会记录本机的操作，方便和主服务器进行对比数据是否同步还可以用于错误恢复。
        2. 如果集群中大部分服务器down机了，保留活着的节点都为 secondary状态并停止，不选举了。
        3. 如果集群中选举出来的主节点或者所有从节点最后一次同步时间看起来很旧了，停止选举等待人来操作。
        4. 如果上面都没有问题就选择最后操作时间戳最新（保证数据是最新的）的服务器节点作为主节点。

        自动故障转移

        ​ 如果主节点未与配置的其他节点通信超过配置的electionTimeoutMillis时间段（默认为10秒），则符合条件，其他的从节点讲要求选举将自己指定为新的主要设备。群集尝试完成新主节点的选举并恢复正常操作。

        ​ 在选举成功之前，副本集是不能进行写操作的，但是如果查询操作已经被配置可以运行在从节点上，当主节点挂掉时副本集还是能继续为读操作提供服务的。

        ​ 默认的集群配置下，群集选择新主节点通常不应超过12秒的时间，这包括将原来挂掉的主节点标记为不可用状态以及请求并完成新主节点选举的时间，可以通过修改副本集配置选项electionTimeoutMillis来调整此时间段 。网络延迟等因素可能会延长副本集选举完成所需的时间，从而影响群集在没有主节点的情况下运行的时间。这些因素取决于您的特定群集体系结构。

        节点部署策略

        1. 如果复制集最多可以有50个节点，只能有7个投票节点，再新增的节点必须设置为non-voting

        2. 保证复制集有奇数个投票节点，如果有已经有偶数个投票节点了，那么可以再部署一个仲裁者节点

        3. 添加隐藏节点可以实现一些特定的功能，比如备份

        4. 根据读的情况，是否开启从节点的读处理

        5. 始终保证有一个备份节点，可以选择磁盘空间比较大的

        # 配置说明

        配置方法：

        • rs.initiate()初始化复制集
        • rs.reconfig()覆盖更新复制集的配置

        配置信息说明：

        • _id：同配置文件中的复制集名称保持一致
        • version：版本
        • protocolVersion：默认为1，从4.0版本开始，只能是1，不能设置为0
        • writeConcernMajorityJournalDefault：默认为true
        • configsvr：默认为false，复制集是否用于集群
        • members._id：复制集中该节点的唯一标识
        • members.host：该节点的域名或IP地址+端口号，每个节点之间必须可以通信
        • members.arbiterOnly：默认为false，是否为仲裁者节点
        • members.buildIndexes：默认为true，节点是否创建索引，对于要处理读操作的节点最好设置成true
        • members.hidden：默认为false，是否为隐藏节点
        • members.priority：优先级，主节点和从节点的数值范围为0到1000，仲裁者节点的数值只能是0或1；表示成为主节点的优先级，数值越大表示成为主节点的优先级越高；该值不为0时，则votes不能为0
        • members.slaveDelay：默认为0；用于创建延时节点，数据相对于主节点滞后的秒数；即数据比主节点陈旧的多少时间，可以用来作为数据修复时的数据备份
        • members.votes：默认为1，值的范围为1或0；等于1时表示该节点有投票权限，否则没有投票权限
        • settings.chainingAllowed：默认为true，是否允许从节点从其他从节点同步数据；为false时则表示从节点只能从主节点同步数据
        • settings.heartbeatTimeoutSecs：默认为10；判定节点失联的描述，如果在这个时间只能没有得到相应节点的反馈，则认为失联
        • settings.electionTimeoutMillis：默认为10000(10秒)；判定主节点失联的时间间隔，即如果在该时间段内没有检测到主节点的心跳，则发起选举
        • settings.catchUpTimeoutMillis：默认为-1，新选出来的主节点与有更加新数据节点之间的时间限制；无限或高时间限制可能会减少其他成员在选举后需要回滚的数据量，但可能会增加故障转移时间。
        • settings.replicaSetId：不能改变该参数的值，配置的时候不要去动即可

        # 节点操作

        1. 新增节点：rs.add(host, arbiterOnly) host内容可以对照上面结构
        2. 只添加仲裁者节点：rs.addArb(host)
        3. rs.freeze(秒数)：只能在从节点执行，在某一时间段之内，该节点无法成为主节点
        4. rs.printReplicationInfo()：输出oplog的相关信息
        5. rs.printSlaveReplicationInfo()：输出从节点oplog的信息
        6. rs.remove(hostname)：移除节点，比如rs.remove("127.0.0.1:27017")
        7. db.getMongo().setSlaveOk(false)：在从节点运行，禁止在从节点运行读操作；改成true为允许
        8. rs.slaveOk()：在从节点执行，允许读操作的简写
        9. rs.syncFrom(host)：从哪一个节点同步数据，比如rs.syncFrom("127.0.0.1:27017")有投票权限的节点，不能从没有投票权限的节点同步数据
        10. rs.stepDown(stepDownSecs, secondaryCatchUpPeriodSecs)：将主节点降级为从节点，该命令不会立马生效；如果没有从节点被选出来成为主节点，则等待第二个参数的时间选出一个从节点成为主节点。一但选出，这主节点里面降级为从节点：
        • stepDownSecs：将主节点降级的秒数，如果不填，则默认为60秒
        • secondaryCatchUpPeriodSecs：默认为10秒

        ###其他

        其实不只是主节点、副本节点、仲裁节点，还有Secondary-Only、Hidden、Delayed、Non-Voting。

        **Secondary-Only:**不能成为primary节点，只能作为secondary副本节点，防止一些性能不高的节点成为主节点。 **Hidden:**这类节点是不能够被客户端制定IP引用，也不能被设置为主节点，但是可以投票，一般用于备份数据。 **Delayed：**可以指定一个时间延迟从primary节点同步数据。主要用于备份数据，如果实时同步，误删除数据马上同步到从节点，恢复又恢复不了。 **Non-Voting：**没有选举权的secondary节点，纯粹的备份数据节点。

        # 分片集群

        ###概述

        ####背景

        ​ 为解决mongodb在replica set每个从节点上面的数据库均是对数据库的全量拷贝，从节点压力在高并发大数据量的场景下存在很大挑战，同时考虑到后期mongodb集群的在数据压力巨大时的扩展性，应对海量数据引出了分片机制。

        # 分片概念

        ​ 分片是将数据库进行拆分，将其分散在不同的机器上的过程，无需功能强大的服务器就可以存储更多的数据，处理更大的负载，在总数据中，将集合切成小块，将这些块分散到若干片中，每个片只负载总数据的一部分，通过一个知道数据与片对应关系的组件mongos的路由进程进行操作。

        # 基础组件

        四个组件：mongos，config server，shard，replica set

        mongos：数据库集群请求的入口，所有请求需要经过mongos进行协调，无需在应用层面利用程序来进行路由选择，mongos其自身是一个请求分发中心，负责将外部的请求分发到对应的shard服务器上，在生产环境通常有多mongos作为请求的入口，防止其中一个挂掉所有的mongodb请求都没有办法操作。

        **config server：**配置服务器，存储所有数据库元数据（分片，路由）的配置。mongos本身没有物理存储分片服务器和数据路由信息，只是存缓存在内存中来读取数据，mongos在第一次启动或后期重启时候，就会从config server中加载配置信息，如果配置服务器信息发生更新会通知所有的mongos来更新自己的状态，从而保证准确的请求路由，生产环境中通常也需要多个config server，防止配置文件存在单节点丢失问题。从3.4版本开始，config server必须是复制集的模式。

        **shard：**在传统意义上来讲，如果存在海量数据，单台服务器存储1T压力非常大，无论考虑数据库的硬盘，网络IO，又有CPU，内存的瓶颈，如果多台进行分摊1T的数据，到每台上就是可估量的较小数据，在mongodb集群只要设置好分片规则，通过mongos操作数据库，就可以自动把对应的操作请求转发到对应的后端分片服务器上。从3.6版本开始，每一个分片必须是复制集的模式。

        **replica set：**在总体mongodb集群架构中，对应的分片节点，如果单台机器下线，对应整个集群的数据就会出现部分缺失，这是不能发生的，因此对于shard节点需要replica set来保证数据的可靠性，生产环境通常为2个副本+1个仲裁。

        ####架构图

        # 存储说明

        1. shard keys：分片key，类似于mycat中的分库键：
        • 决定数据存在哪一个分片
        • 可以由一个字段或多个字段组成
        • 一旦创建，不能更改
        • 必须为组成分片key的字段创建索引，否则mongo会进行自动创建
        • Range sharding：基于范围的分片
          1. 数据根据键值线性分部，所以key值相近的数据可能会存在一个chunk中
          2. 数据文件挪动小
        • Hash sharding：基于key值进行hash计算进行数据分布
          1. 数据分部比较均匀，写数据的压力可以分摊到各个节点
          2. 会有大量的随机IO

        1. 不做分片的collection，数据都存储在主分片中

        2. Chunk：数据块

        • 在分片集群中数据会划分存储在一个一个的chunk中，每一个chunk都存着一部分数据
        • Chunk的初始大小为64M，如果超过这个大小，会进行拆分，防止过大
        • 数据更新或者新增可能触发chunk的拆分动作，拆分时不会影响访问
        • Chunk的拆分和移动会消耗IO资源，设置的大小会影响拆分、迁移的资源消耗

        # mongodb应用场景

        ###适用场景

        根据MongoDB 官网的说明，MongoDB 的适用场景如下:

        1. 网站实时数据：mongoDB非常适合实时的插入，更新与查询，并具备网站实时数据存储所需的复制及高度伸缩性。

        2. 数据缓存:由于性能很高，MongoDB 也适合作为信息基础设施的缓存层。在系统重启之后，由MongoDB搭建的持久化缓存层可以避免下层的数据源过载。

        3. 大尺寸、低价值数据存储:使用传统的关系型数据库存储一些数据时可能会比较昂贵，在此之前，很多时候程序员往往会选择传统的文件进行存储。

        4. 高伸缩性场景:MongoDB 非常适合由数十或数百台服务器组成的数据库。MongoDB 的路线图中已经包含对MapReduce 引擎的内置支持。

        5. 对象或JSON 数据存储:MongoDB 的BSON 数据格式非常适合文档化格式的存储及查询。

        # 不适合场景

        1. 高度事务性系统:例如银行或会计系统。传统的关系型数据库目前还是更适用于需要大量原子性复杂 事务的应用程序。

        2. 传统的商业智能应用:针对特定问题的BI 数据库会对产生高度优化的查询方式。对于此类应用， 数据仓库可能是更合适的选择。

        3. 需要复杂SQL 查询的问题。
